# Daily Log and Build Log: Linked List Implementation

This log describes the work completed during the Linked List implementation. The day is divided into four working sessions of two hours each, from **10:00 AM to 6:00 PM**.

---

# Daily Log

## Session 1: 10:00 AM - 12:00 PM

During the first session, the basic structure of the Linked List module was planned and initialized. The main focus was on understanding the required operations and deciding how the data structure would be represented internally.

The `Node<T>` structure was designed to store two main parts: the data value and the pointer to the next node. The `LinkedList<T>` class was also planned with two private members: `head`, which points to the first node, and `size`, which stores the current number of nodes.

The template-based design was selected so that the linked list could store different data types without rewriting the same logic for each type.

**Work Completed:**

- Planned the Linked List class structure.
- Designed the `Node<T>` representation.
- Added `head` and `size` as internal data members.
- Decided to use templates for generic implementation.
- Identified the required public functions.

---

## Session 2: 12:00 PM - 2:00 PM

During the second session, the basic node insertion logic was implemented. The focus was on adding new nodes correctly and maintaining proper links between nodes.

Special handling was added for the empty-list case. If the list is empty, the newly inserted node becomes the head. For insertion at other positions, pointer links are updated carefully so that the list remains connected.

The `size` variable was updated after every successful insertion so that the current number of nodes could be returned in constant time.

**Work Completed:**

- Implemented basic node insertion logic.
- Handled insertion at required positions.
- Handled insertion in an empty list.
- Updated the `size` variable after insertion.
- Checked pointer linking for newly inserted nodes.

---

## Session 3: 2:00 PM - 4:00 PM

During the third session, deletion and indexed operations were implemented. The `pop_front()` function was added to remove the first node, and the `pop_back()` function was added to remove the last node.

The indexed functions `insertAtIndex()` and `deleteAtIndex()` were also implemented. Boundary cases such as insertion or deletion at the first position were handled separately to keep pointer updates correct.

Exception handling was added for invalid indexes and deletion from an empty list. Invalid index access should throw `std::out_of_range`, while deletion from an empty list should throw `std::underflow_error`.

**Work Completed:**

- Implemented `pop_front()`.
- Implemented `pop_back()`.
- Implemented `insertAtIndex(int index, T val)`.
- Implemented `deleteAtIndex(int index)`.
- Added index validation.
- Added exception handling for invalid and empty-list operations.

---

## Session 4: 4:00 PM - 6:00 PM

During the final session, utility functions and memory management logic were completed. The `get()` function was implemented to return a reference to the element at a given index. The `search()` function was added to find the first matching value, and the `traverse()` function was added to display all elements.

The destructor, copy constructor, and copy assignment operator were reviewed as part of the Rule of Three. This ensures that the linked list manages dynamically allocated memory correctly and avoids memory leaks, shallow copies, dangling pointers, and double deletion.

Basic testing and review were performed to verify insertion, deletion, traversal, searching, and size updates.

**Work Completed:**

- Implemented `get(int index)`.
- Implemented `search(T val)`.
- Implemented `traverse()`.
- Implemented or reviewed `getSize()`.
- Reviewed Rule of Three requirements.
- Checked memory cleanup and deep-copy behavior.
- Performed basic functional testing.
